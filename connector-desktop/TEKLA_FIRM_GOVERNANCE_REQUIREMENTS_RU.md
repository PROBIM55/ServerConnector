# TEKLA_FIRM_GOVERNANCE_REQUIREMENTS_RU

## 1. Цель

Зафиксировать целевую модель управления стандартом Tekla (`XS_FIRM`) в текущей инфраструктуре, включая:

- место хранения эталона;
- разграничение ролей;
- пользовательские сценарии администратора;
- публикацию изменений через интерфейс `Structura Connector`;
- rollout изменений пользователям.

## 2. Эталон и хранилища

### Эталонная папка фирмы

- Единый эталон хранится по пути:
  - `\\62.113.36.107\BIM_Models\Tekla\XS_FIRM`

### Git-репозиторий

- Используется текущий репозиторий коннектора.
- Для стандарта Tekla выделяется отдельный каталог в репозитории:
  - `tekla-firm-standard/`

## 3. Роли и разграничение прав

Нужны две независимые роли:

1. `admin_system`
   - управление токенами, сетевым доступом, сервисными операциями в админке.

2. `admin_firm`
   - управление стандартом Tekla (`XS_FIRM`), публикация изменений в git, выпуск версий стандарта.

Требования к ролям:

- роли назначаются независимо;
- один пользователь может иметь обе роли;
- пользователь без `admin_firm` не может публиковать изменения стандарта.

## 4. Политика доступа к SMB

### Для `admin_firm`

- `Read/Write` на `\\62.113.36.107\BIM_Models\Tekla\XS_FIRM`.

### Для остальных пользователей

- `Read-Only` на `\\62.113.36.107\BIM_Models\Tekla\XS_FIRM`.

## 5. UX и функционал в интерфейсах

## 5.1 Admin UI (web)

Админка должна уметь:

- назначать/снимать роль `admin_firm` пользователям;
- показывать список пользователей с ролью `admin_firm`;
- писать аудит событий назначения/снятия роли.

## 5.2 Desktop Connector

### Для всех пользователей

- вкладка `Стандарт Tekla` (статус, версия, проверка/обновление, лог).

### Только для `admin_firm`

- дополнительный раздел управления эталоном `XS_FIRM`:
  - просмотр измененных файлов;
  - ввод комментария публикации;
  - действие `commit + push` через UI;
  - результат операции и лог.

## 6. Операционный процесс admin_firm

1. Вносит изменения в эталонную папку `XS_FIRM`.
2. В `Structura Connector` (режим admin_firm) выполняет:
   - проверку изменений,
   - подготовку публикации,
   - commit/push в git.
3. После push публикует версию стандарта (manifest version/revision/repo_ref).
4. Клиенты получают обновление через существующий механизм Tekla Adapter.

## 7. Git-процесс (через Connector)

Обязательные требования:

- используется только встроенный git (без зависимости от системного PATH);
- push разрешен только при роли `admin_firm`;
- все publish-операции пишутся в аудит;
- должна быть защита от параллельной публикации (single publish lock).

Минимальный pipeline публикации:

1. `fetch/rebase` (или эквивалент);
2. commit с комментарием;
3. push;
4. запись `commit hash`, `actor`, `timestamp` в аудит.

## 8. Поведение для обычных пользователей

- не пушат изменения в git;
- не редактируют эталонную `XS_FIRM`;
- получают обновления стандарта автоматически в локальную папку:
  - `C:\Company\TeklaFirm`.

## 9. Инварианты системы

- Единый эталон: `\\62.113.36.107\BIM_Models\Tekla\XS_FIRM`.
- Публикация стандарта - только роль `admin_firm`.
- Распространение стандарта пользователям - только через Connector.
- `admin_system` и `admin_firm` разделены.

## MVP (обязательный первый этап)

### A. Governance и роли

- Добавить в серверную модель пользователей отдельный флаг/роль `admin_firm`.
- Добавить API для назначения/снятия `admin_firm`.
- Добавить отображение роли в admin UI.

### B. Доступ к эталону

- Зафиксировать эталонный SMB-путь `\\62.113.36.107\BIM_Models\Tekla\XS_FIRM`.
- Внедрить политику ACL:
  - `admin_firm` -> RW
  - остальные -> RO

### C. Desktop (admin_firm mode)

- Добавить в desktop UI режим/секцию `Управление XS_FIRM`.
- Показывать секцию только при `admin_firm`.
- Реализовать:
  - `Проверить изменения`
  - `Подготовить публикацию`
  - `Опубликовать (commit+push)`

### D. Аудит

- Логировать:
  - назначение/снятие `admin_firm`
  - commit/push публикации стандарта
  - публикацию manifest версии

### E. Acceptance MVP

- Админ системы назначает `admin_firm` пользователю.
- Пользователь с `admin_firm` видит секцию публикации в desktop.
- Пользователь без `admin_firm` секцию не видит.
- Публикация commit+push выполняется через UI и попадает в аудит.

## Phase 2 (расширение)

### A. Управляемый rollout

- Поддержка пилотной группы для новой версии стандарта.
- Массовый rollout после пилота.
- Отдельный статус rollout в admin UI.

### B. Безопасность и надежность публикации

- Блокировка параллельных публикаций.
- Проверка чистоты рабочей копии перед публикацией.
- Стандартизованный шаблон commit message.

### C. Дополнительная эксплуатация

- Откат через выбор предыдущего `repo_ref` из UI.
- Расширенная диагностика ошибок публикации.
- Автоматическая проверка консистентности эталонной папки.

### D. Acceptance Phase 2

- Возможен контролируемый pilot rollout и rollback.
- Админка показывает полный статус rollout по пользователям.
- Ошибки публикации диагностируются из UI без ручного доступа к серверу.
