# Чеклист реализации адаптера стандартов Tekla в Structura Connector

Документ фиксирует полный список работ для реализации механизма локальной папки фирмы Tekla с централизованным обновлением через `Structura Connector`.

## 1. Базовые архитектурные решения

- [x] Зафиксировать единый локальный путь папки фирмы для всех ПК, например `C:\Company\TeklaFirm`.
- [x] Зафиксировать, что пользователи не редактируют локальную папку фирмы вручную.
- [x] Зафиксировать, что `Structura Connector` использует встроенный git-компонент, а не системный `git`.
- [x] Зафиксировать, что основной сценарий обновления - инкрементальный, без полной перезаливки всей папки.
- [x] Зафиксировать, что проверка обновлений выполняется через легкий `manifest`, а git используется только при наличии новой версии.

## 2. Репозиторий эталонной папки фирмы

- [x] Создать отдельный git-репозиторий для эталонной `firm folder`.
- [x] Определить структуру папок в репозитории.
- [x] Подготовить `.gitignore` для исключения мусора и локальных временных файлов.
- [x] Описать правила публикации изменений администратора.
- [x] Зафиксировать модель релизов: теги, ветка публикации или стабильная ветка.
- [x] Описать порядок тестирования изменений перед публикацией пользователям.

## 3. Серверная публикация версий

- [x] Определить, где будет публиковаться `manifest` актуальной версии стандарта.
- [x] Определить формат `manifest`.
- [x] Включить в `manifest` минимум следующие поля:
  - [x] `version`
  - [x] `revision`
  - [x] `published_at`
  - [x] `target_path`
  - [x] `minimum_connector_version`
  - [x] `notes`
- [x] Реализовать endpoint или URL для чтения актуального `manifest`.
- [x] Реализовать URL/источник для доступа клиента к git-источнику или к опубликованной git-ревизии.
- [x] Зафиксировать режим публикации: только стабильные версии доступны клиентам.

## 4. Встроенный git-компонент в коннекторе

- [x] Выбрать bundled/portable git для включения в дистрибутив коннектора.
- [x] Определить путь размещения git внутри установленного приложения.
- [x] Реализовать вызов встроенного `git.exe` из кода коннектора.
- [x] Исключить зависимость от системного PATH.
- [x] Добавить проверку доступности встроенного git при запуске коннектора.
- [x] Добавить диагностический лог по встроенному git.

## 5. Инициализация локальной папки фирмы

- [x] Реализовать создание папки `C:\Company\TeklaFirm`, если она отсутствует.
- [x] Реализовать первичную инициализацию локальной git-копии.
- [x] Реализовать первый `clone` или эквивалентный bootstrap.
- [x] Реализовать запись локального служебного состояния после первой установки.
- [x] Проверить, что структура локальной папки соответствует ожидаемой структуре `firm folder`.

## 6. Локальное состояние и служебные данные

- [x] Определить формат локального state-файла.
- [x] Хранить локально:

  - [x] текущую локальную ревизию
  - [x] целевую опубликованную ревизию
  - [x] дату последней проверки
  - [x] дату последнего успешного обновления
  - [x] флаг отложенной установки после закрытия Tekla
- [x] Реализовать безопасную перезапись state-файла.

## 7. Легкая проверка обновлений

- [x] Реализовать получение server-side `manifest`.
- [x] Реализовать сравнение локальной версии и серверной версии.
- [x] Реализовать режим легкой проверки без запуска `git fetch`, если версия не изменилась.
- [x] Реализовать проверку обновлений при запуске коннектора.
- [x] Реализовать периодическую проверку обновлений по расписанию, например раз в 30 минут.
- [ ] Убедиться, что фоновая проверка почти не нагружает CPU, RAM и сеть.

## 8. Git-обновление по требованию

- [x] Реализовать запуск git-обновления только если `manifest` показал новую версию.
- [x] Реализовать обновление локальной папки фирмы до целевой ревизии.
- [x] Убедиться, что обновляются только измененные файлы.
- [x] Убедиться, что новые файлы добавляются.
- [x] Убедиться, что удаленные из стандарта файлы удаляются локально.
- [x] Реализовать обработку ошибок git-операций.

## 9. Обнаружение запущенной Tekla

- [x] Реализовать определение, запущен ли процесс Tekla.
- [x] Реализовать функцию проверки перед применением обновления.
- [x] Убедиться, что обновление не стартует автоматически при работающей Tekla.
- [x] Реализовать служебный флаг `update_pending_after_tekla_close`.

## 10. Логика установки обновления

- [x] Если Tekla не запущена, выполнять обновление автоматически без запроса.
- [x] Если Tekla запущена, не выполнять обновление сразу.
- [x] Если Tekla запущена, переводить состояние в `Ожидает установки после закрытия Tekla`.
- [x] Если Tekla была запущена, а затем закрыта, выполнять отложенное обновление при следующей проверке автоматически.
- [x] Реализовать ручную кнопку `Обновить сейчас`, если Tekla уже закрыта.

## 22. Критерий завершения

Реализацию можно считать завершенной, если:

- [x] пользователь устанавливает только `Structura Connector`;
- [x] локальная папка фирмы создается и поддерживается автоматически;
- [ ] одинаковый `XS_FIRM` используется на всех рабочих местах;
- [ ] при отсутствии новой версии фоновая работа почти не нагружает ПК;
- [x] при наличии новой версии обновляются только измененные файлы;
- [x] при запущенной Tekla обновление корректно откладывается;
- [x] после закрытия Tekla обновление применяется автоматически;
- [x] пользователь видит понятный статус и журнал;
- [x] администратор может публиковать новые версии стандарта управляемо.
