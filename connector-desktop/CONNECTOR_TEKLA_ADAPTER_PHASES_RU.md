# Phased Roadmap реализации адаптера стандартов Tekla

Документ разбивает реализацию адаптера стандартов Tekla в `Structura Connector` на этапы, чтобы задачу можно было отдавать в разработку по приоритетам.

Связанный полный чеклист:

- `connector-desktop/CONNECTOR_TEKLA_ADAPTER_CHECKLIST_RU.md`

## Phase 1 - Технический фундамент

Цель этапа:

- подготовить архитектурную основу;
- встроить git-компонент;
- обеспечить локальную папку фирмы;
- реализовать легкую проверку версий без тяжелой фоновой нагрузки.

### Что должно быть реализовано

- [ ] Зафиксирован единый путь `C:\Company\TeklaFirm`.
- [ ] В коннектор встроен bundled/portable git.
- [ ] Коннектор умеет определять путь встроенного `git.exe`.
- [ ] Реализован первичный bootstrap локальной папки фирмы.
- [ ] Реализован локальный state-файл.
- [ ] Реализован server-side `manifest`.
- [ ] Коннектор умеет читать `manifest`.
- [ ] Коннектор умеет сравнивать локальную и серверную версии.
- [ ] Реализована легкая проверка при запуске.
- [ ] Реализована периодическая легкая проверка, например раз в 30 минут.
- [ ] При отсутствии новой версии коннектор не запускает тяжелые git-операции.

### Результат этапа

- у клиента появляется база для controlled update;
- фоновая проверка почти не нагружает ПК;
- можно видеть, что локально установлено и что опубликовано на сервере.

## Phase 2 - Рабочее обновление локальной папки фирмы

Цель этапа:

- сделать реальное инкрементальное обновление локальной `firm folder`;
- обеспечить безопасное применение обновления только при закрытой Tekla.

### Что должно быть реализовано

- [ ] Реализован git-clone локальной папки фирмы.
- [ ] Реализован переход к целевой опубликованной ревизии.
- [ ] Обновляются только измененные файлы.
- [ ] Новые файлы добавляются автоматически.
- [ ] Удаленные из стандарта файлы удаляются локально.
- [ ] Обновляется локальное состояние после успешного update.
- [ ] Реализовано определение, запущена ли Tekla.
- [ ] Если Tekla не запущена, обновление применяется автоматически.
- [ ] Если Tekla запущена, обновление не выполняется сразу.
- [ ] Реализован флаг `Ожидает установки после закрытия Tekla`.

### Результат этапа

- коннектор уже умеет обновлять локальную папку фирмы без ручной установки git;
- обновление не ломает активную сессию Tekla.

## Phase 3 - Пользовательский интерфейс и UX

Цель этапа:

- сделать функцию понятной для обычного пользователя;
- добавить отдельную вкладку и прозрачные статусы.

### Что должно быть реализовано

- [ ] Добавлена отдельная вкладка `Стандарт Tekla`.
- [ ] На вкладке видны:
  - [ ] локальная версия
  - [ ] серверная версия
  - [ ] путь локальной папки
  - [ ] текущий статус
  - [ ] дата последней проверки
  - [ ] дата последнего успешного обновления
- [ ] Реализованы кнопки:
  - [ ] `Проверить обновление`
  - [ ] `Обновить сейчас`
  - [ ] `Открыть папку`
  - [ ] `Открыть журнал`
- [ ] Реализованы статусы:
  - [ ] `Актуально`
  - [ ] `Доступно обновление`
  - [ ] `Ожидает установки после закрытия Tekla`
  - [ ] `Обновление выполняется`
  - [ ] `Ошибка обновления`
- [ ] Реализовано уведомление из трея, если Tekla запущена и доступно обновление.

### Результат этапа

- пользователь понимает текущее состояние стандарта Tekla;
- у пользователя есть понятный UI и прозрачные статусы.

## Phase 4 - Автоматизация отложенного обновления

Цель этапа:

- довести сценарий до полностью автоматической ежедневной работы без участия пользователя в обычных случаях.

### Что должно быть реализовано

- [ ] Если Tekla была запущена, update переводится в pending.
- [ ] При следующей проверке, если Tekla уже закрыта, update ставится автоматически.
- [ ] Pending-state сохраняется между перезапусками коннектора.
- [ ] Реализован понятный локальный журнал операций.
- [ ] Реализованы user-friendly сообщения об ошибках.

### Результат этапа

- пользователь редко делает что-то вручную;
- обновление в большинстве случаев происходит автоматически и безопасно.

## Phase 5 - Администрирование и rollout

Цель этапа:

- дать администратору управляемый процесс публикации стандартов Tekla и контроль состояния клиентов.

### Что должно быть реализовано

- [ ] Зафиксирована процедура публикации новой версии стандарта.
- [ ] Зафиксирована процедура тестирования на пилотных пользователях.
- [ ] Реализован журнал версий стандарта.
- [ ] Подготовлен rollback-сценарий.
- [ ] В админке Connector отображается текущая опубликованная версия.
- [ ] В админке Connector виден статус клиентов:
  - [ ] кто обновился
  - [ ] кто не обновился
  - [ ] у кого update pending
  - [ ] у кого ошибка
- [ ] Реализован аудит публикаций и клиентских обновлений.

### Результат этапа

- администратор управляет rollout по команде централизованно;
- можно безопасно публиковать, контролировать и откатывать версии.

## Phase 6 - Надежность и эксплуатация

Цель этапа:

- довести решение до production-качества.

### Что должно быть реализовано

- [ ] Реализована диагностика встроенного git.
- [ ] Реализовано восстановление после неуспешного обновления.
- [ ] Реализована проверка целостности локальной папки после update.
- [ ] Реализована резервная копия или rollback на предыдущую ревизию.
- [ ] Проверена нагрузка на CPU/RAM/сеть.
- [ ] Подготовлен runbook диагностики.
- [ ] Подготовлена пользовательская инструкция.
- [ ] Подготовлена инструкция для администратора по публикации версий.

### Результат этапа

- решение готово к долгосрочной эксплуатации;
- команда понимает, как его поддерживать и диагностировать.

## Рекомендуемый порядок запуска в работу

### MVP

Для первого рабочего релиза достаточно закрыть:

- Phase 1
- Phase 2
- Phase 3
- Phase 4

Это даст:

- встроенный git;
- локальную папку фирмы;
- отдельную вкладку `Стандарт Tekla`;
- легкую фоновую проверку;
- безопасное инкрементальное обновление;
- отложенную установку при открытой Tekla.

### Следующий шаг после MVP

После MVP логично закрывать:

- Phase 5

Это даст централизованный rollout и контроль со стороны администратора.

### Финальный эксплуатационный этап

Затем закрывается:

- Phase 6

Это переводит решение в стабильный production-режим.

## Практический приоритет задач

Если отдавать это разработчику или агенту, рекомендуемый приоритет такой:

1. встроенный git + локальный путь + state
2. manifest-check
3. git-update при новой версии
4. определение процесса Tekla
5. вкладка `Стандарт Tekla`
6. tray-уведомления и pending-логика
7. журнал и диагностика
8. админский rollout и аудит
