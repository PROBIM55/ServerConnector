# Tekla Standard - гайд администратора по публикации

## Цель

Документ описывает процесс публикации новой версии стандарта Tekla для клиентов `Structura Connector`.

## Предусловия

- Эталонная папка фирмы ведется в отдельном git-репозитории.
- В admin UI доступен блок `Стандарт Tekla (публикация)`.
- Endpoint сервера доступен: `/updates/tekla/firm/latest.json`.

## Формат релиза

Перед публикацией зафиксировать:

- `version` - человекочитаемая версия (например `2026.03.2`)
- `revision` - ревизия пакета (например `r3` или короткий commit)
- `repo_url` - URL git-репозитория стандарта
- `repo_ref` - ветка/тег для клиентского checkout (например `main` или `v2026.03.2`)
- `target_path` - локальный путь на клиентах (`C:\Company\TeklaFirm`)
- `minimum_connector_version` - минимальная версия клиента

## Публикация версии

1. Проверить, что изменения в репозитории стандарта готовы к rollout.
2. Открыть admin UI -> блок `Стандарт Tekla (публикация)`.
3. Заполнить поля версии/ревизии/repo.
4. Нажать `Опубликовать версию Tekla`.
5. Убедиться, что верхний блок manifest показывает новую версию.
6. Проверить, что в аудите появилось событие `tekla_manifest_updated`.

## Пилот и массовый rollout

### Пилот

1. Выбрать 1-2 тестовых пользователя.
2. Убедиться, что у них коннектор обновил `installed_revision` до `target_revision`.
3. Проверить открытие Tekla и базовый рабочий сценарий.

### Массовый rollout

1. После успешного пилота оповестить команду.
2. Дождаться автоматического обновления по расписанию или попросить нажать `Проверить обновление`.
3. В admin UI контролировать:
   - `кто обновился`
   - `у кого pending`
   - `у кого ошибка`

## Откат

Если обнаружена проблема:

1. Вернуть `repo_ref` на предыдущую стабильную ревизию.
2. Опубликовать manifest повторно с откатной версией.
3. Проконтролировать, что клиенты вернулись к предыдущей ревизии.

## Журнал версий

Рекомендуется вести отдельный changelog-файл в репозитории стандарта с полями:

- версия
- ревизия
- дата публикации
- ключевые изменения
- результат пилота
