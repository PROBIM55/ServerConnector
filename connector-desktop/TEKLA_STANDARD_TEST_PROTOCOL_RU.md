# Tekla Standard - протокол тестирования (E2E)

## Цель

Проверить рабочий цикл обновления стандарта Tekla через `Structura Connector` на реальных клиентских ПК.

## Тестовая среда

- Сервер с актуальным `tekla_firm_latest.json`
- Admin UI доступен
- Минимум 2 клиентских ПК
- На клиентах установлен новый `Structura Connector`

## Набор тестов

## T1 - Первая инициализация

1. На чистом ПК запустить коннектор.
2. Открыть вкладку `Стандарт Tekla`.
3. Нажать `Проверить обновление`.
4. Убедиться, что локальная папка `C:\Company\TeklaFirm` создается.
5. Проверить, что `installed_revision` заполнена.

Ожидаемо:

- статус без ошибки;
- локальная папка создана;
- лог содержит запись об установке.

## T2 - Нет новой версии

1. Опубликованная версия = установленной.
2. Нажать `Проверить обновление`.

Ожидаемо:

- статус `Актуально`;
- git-операции обновления не выполняются.

## T3 - Новая версия, Tekla закрыта

1. Опубликовать новую ревизию в admin UI.
2. На клиенте убедиться, что Tekla не запущена.
3. Нажать `Проверить обновление` (или дождаться цикла).

Ожидаемо:

- обновление применяется автоматически;
- `installed_revision == target_revision`;
- `pending_after_close = false`.

## T4 - Новая версия, Tekla запущена

1. Открыть Tekla на клиенте.
2. Опубликовать новую ревизию.
3. На клиенте выполнить проверку.

Ожидаемо:

- статус `Ожидает установки после закрытия Tekla`;
- есть уведомление из трея;
- обновление не применяется сразу.

## T5 - Отложенное применение

1. После T4 закрыть Tekla.
2. Не нажимать ручное обновление.
3. Дождаться следующего цикла проверки.

Ожидаемо:

- обновление применяется автоматически;
- pending-флаг снимается;
- в admin UI клиент переходит в обновленный.

## T6 - Ошибка git/репозитория

1. Временно опубликовать невалидный `repo_url` или `repo_ref`.
2. Запустить проверку.

Ожидаемо:

- статус `Ошибка обновления`;
- заполнен `last_error`;
- в admin UI клиент отображается в ошибках.

## T7 - Откат

1. Опубликовать предыдущий стабильный `repo_ref`.
2. Запустить проверку.

Ожидаемо:

- клиент возвращается на предыдущую ревизию;
- ошибка исчезает.

## T8 - Нагрузка

1. На клиенте в период без новых версий замерить нагрузку.
2. Проверить CPU/RAM/сеть во время легкой проверки.

Ожидаемо:

- заметной нагрузки нет;
- тяжелые git-операции не запускаются без новой версии.

## Точки контроля на сервере

- `GET /updates/tekla/firm/latest.json`
- `GET /admin/tekla/clients`
- Аудит событий:
  - `tekla_manifest_updated`
  - `tekla_client_state`

## Критерий приемки

Решение считается принятым, если T1-T8 проходят без критических дефектов на пилотной группе.
