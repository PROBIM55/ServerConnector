# План развития Structura Connector как адаптера стандартов Tekla

## Цель

Расширить `Structura Connector` так, чтобы он управлял локальной корпоративной папкой фирмы Tekla у каждого пользователя и централизованно поддерживал ее в актуальном состоянии.

Ключевая идея:

- пользователи не работают с сетевой `firm folder`;
- у всех пользователей используется одинаковый локальный путь;
- содержимое локальной папки обновляется инкрементально;
- пользователь не устанавливает дополнительные инструменты вручную.

## Исходные условия

Для текущей инфраструктуры зафиксированы следующие правила:

- сетевая `firm folder` через VPN/медленный SMB не подходит;
- локальные изменения пользователей в папке фирмы не допускаются;
- у всех пользователей должен быть одинаковый путь `XS_FIRM`;
- обновление должно происходить через коннектор;
- отдельная ручная установка `git` пользователем недопустима.

## Целевая схема

### 1. Локальный путь у пользователя

На всех рабочих местах используется один и тот же путь:

- `C:\Company\TeklaFirm`

Это позволяет использовать один и тот же путь `XS_FIRM` у всех участников команды.

### 2. Источник правды

Эталонная папка фирмы ведется администратором в git-репозитории.

Git используется как:

- источник версий;
- механизм инкрементального обновления;
- способ отката;
- журнал изменений корпоративного стандарта.

### 3. Что получает клиент

На ПК пользователя коннектор работает не с архивом полной папки, а с локальной git-копией папки фирмы.

Содержимое обновляется только по изменившимся файлам.

### 4. Git-компонент

В `Structura Connector` встраивается служебный git-компонент.

Это означает:

- пользователь не ставит `Git for Windows` вручную;
- коннектор использует встроенный `git.exe`;
- версия git фиксируется и поставляется вместе с коннектором.

Пример расположения встроенного инструмента:

- `C:\Program Files\Structura Connector\tools\git\bin\git.exe`

## Почему выбран git-подход

Полная переустановка всей `firm folder` в виде большого zip-пакета не подходит, потому что:

- папка может весить сотни мегабайт;
- при частых обновлениях это дает лишний трафик;
- пользовательская машина будет скачивать слишком много повторяющихся данных.

Git-подход лучше, потому что:

- скачиваются только измененные файлы;
- сохраняется история версий;
- можно жестко привязать пользователей к стабильным тегам/релизам;
- проще поддерживать controlled rollout.

## Как коннектор будет понимать, что нужно обновить

Коннектор работает с локальной git-копией `firm folder`.

Сценарий проверки:

1. Коннектор определяет текущую локальную ревизию папки фирмы.
2. Коннектор получает актуальную опубликованную ревизию с сервера.
3. Если ревизии совпадают - обновление не требуется.
4. Если серверная ревизия новее - коннектор готовит обновление.
5. При применении обновления встроенный git подтягивает только изменения.

То есть решение о том, какие файлы нужно заменить:

- не вычисляется вручную в коде поштучно;
- это делает git на основе изменений между локальной и целевой ревизией.

В результате:

- измененные файлы обновляются;
- новые файлы добавляются;
- удаленные из стандарта файлы удаляются локально.

## Требование по минимальной нагрузке на ПК

Фоновая работа механизма обновления не должна создавать заметную нагрузку на рабочий компьютер пользователя.

Поэтому проверка обновлений должна быть двухуровневой:

### Легкая фоновая проверка

Коннектор по расписанию выполняет только легкую проверку:

- обращается к маленькому server-side manifest;
- получает опубликованную версию или ревизию;
- сравнивает ее с локально сохраненной версией;
- не запускает тяжелые git-операции, если версия не изменилась.

Такая проверка должна быть практически незаметна по:

- CPU;
- памяти;
- сетевому трафику.

### Git-обновление только по необходимости

Git-операции должны запускаться только если manifest показал новую версию.

То есть:

- обычная периодическая проверка не делает `git fetch` каждый раз;
- `git fetch/pull` выполняется только когда на сервере опубликована новая ревизия;
- инкрементальная загрузка выполняется только в момент реального обновления.

## Рекомендуемый режим проверки

Оптимальный режим для вашей инфраструктуры:

- легкая проверка manifest при запуске коннектора;
- легкая проверка manifest по расписанию, например раз в 30 минут;
- git-обновление только по событию `обнаружена новая версия`.

Такой подход минимизирует:

- постоянную нагрузку на ПК;
- лишний сетевой трафик;
- лишние обращения к репозиторию.

## Неприемлемый вариант

Не рекомендуется делать так:

- на каждой фоновой проверке запускать полноценный `git fetch`;
- часто сканировать всю папку фирмы без причины;
- постоянно пересчитывать содержимое всей папки при отсутствии новой версии.

Такая схема создает лишнюю нагрузку на:

- диск;
- сеть;
- CPU;
- и может ухудшать пользовательский опыт.

## Роль коннектора

Коннектор становится адаптером стандарта Tekla и выполняет следующие функции:

- хранит локальную папку фирмы по фиксированному пути;
- проверяет наличие обновлений;
- обновляет локальную папку инкрементально;
- пишет лог обновлений;
- отслеживает, запущена ли Tekla;
- применяет обновление автоматически, если Tekla не запущена;
- уведомляет пользователя через трей, если Tekla запущена;
- завершает отложенное обновление автоматически при следующей проверке, когда Tekla уже закрыта.

## UI: отдельная вкладка `Стандарт Tekla`

В коннекторе должна появиться отдельная вкладка:

- `Стандарт Tekla`

На этой вкладке отображаются:

- текущая локальная версия;
- доступная версия на сервере;
- путь локальной папки фирмы;
- статус обновления;
- дата последней проверки;
- дата последнего успешного обновления;
- журнал операций.

Кнопки на вкладке:

- `Проверить обновление`
- `Обновить сейчас` (если Tekla не запущена)
- `Открыть папку`
- `Открыть журнал`

## Статусы

Рекомендуемые статусы:

- `Актуально`
- `Доступно обновление`
- `Ожидает установки после закрытия Tekla`
- `Обновление выполняется`
- `Ошибка обновления`

Статус `Ожидает установки после закрытия Tekla` означает:

- новая версия уже обнаружена;
- но Tekla сейчас запущена;
- обновление будет применено автоматически, когда при следующей проверке Tekla уже не будет работать.

## Поведение при проверке обновлений

### Если Tekla не запущена

Коннектор:

- проверяет версию;
- если версия новая, выполняет обновление автоматически без запроса;
- пишет результат в лог;
- обновляет статус в UI.

### Если Tekla запущена

Коннектор:

- не применяет обновление сразу;
- записывает, что доступно обновление;
- показывает уведомление из трея;
- предлагает закрыть Tekla и выполнить обновление вручную.

Если пользователь Tekla закроет, но обновление вручную не запустит:

- на следующей автоматической проверке коннектор увидит, что Tekla уже не работает;
- выполнит обновление автоматически.

## Уведомления из трея

При открытой Tekla нужно показывать понятное уведомление:

- обнаружено обновление папки фирмы Tekla;
- для применения нужно закрыть Tekla;
- после закрытия обновление будет установлено автоматически при следующей проверке или может быть запущено вручную.

На первом этапе не рекомендуется:

- принудительно закрывать Tekla без подтверждения пользователя.

## Логирование

Коннектор должен писать отдельный лог для стандарта Tekla:

- время проверки;
- локальная версия;
- доступная версия;
- была ли запущена Tekla;
- был ли update отложен;
- какие действия выполнены;
- успешно ли завершено обновление;
- текст ошибки при сбое.

## Хранение локального состояния

Локально нужно хранить:

- текущую установленную версию;
- текущую локальную ревизию git;
- целевую опубликованную ревизию;
- дату последней проверки;
- дату последнего успешного обновления;
- флаг, что обновление ожидает установки после закрытия Tekla.

## Что должно быть на сервере

На серверной стороне нужен контур публикации стандартов Tekla.

Минимально необходимое:

- git-репозиторий эталонной `firm folder`;
- опубликованная стабильная ветка или тег;
- manifest с указанием актуальной версии/ревизии;
- endpoint или URL, по которому клиент понимает, какую ревизию считать актуальной.

### Рекомендуемая модель публикации

Админ работает так:

1. меняет эталонную `firm folder`;
2. коммитит изменения в git;
3. помечает стабильную версию тегом или публикуемой веткой;
4. коннектор клиентов обновляется до этой версии.

Это позволяет разделять:

- черновые изменения администратора;
- стабильную версию для пользователей.

## Этапы реализации

## Этап 1. Базовая инфраструктура

Сделать:

- встроенный git-компонент внутри коннектора;
- локальный путь `C:\Company\TeklaFirm`;
- отдельную вкладку `Стандарт Tekla`;
- чтение локальной ревизии;
- чтение опубликованной серверной ревизии;
- ручную кнопку `Проверить обновление`.

Результат:

- можно увидеть, какая версия стандарта установлена локально и какая опубликована на сервере.

## Этап 2. Применение обновлений

Сделать:

- первичный clone локальной папки фирмы;
- автоматическое обновление через встроенный git;
- логирование;
- автообновление при закрытой Tekla.

Результат:

- локальная папка фирмы начинает обновляться инкрементально без участия пользователя.

## Этап 3. Логика работы с Tekla

Сделать:

- проверку, запущена ли Tekla;
- отложенную установку, если Tekla открыта;
- уведомления из трея;
- автоматическое применение при следующей проверке после закрытия Tekla.

Результат:

- обновление не ломает рабочую сессию Tekla и не требует ручного контроля в обычных случаях.

## Этап 4. Администрирование

Добавить в серверную/админскую часть:

- публикацию стабильной версии стандарта;
- просмотр текущей опубликованной версии;
- журнал обновлений по устройствам/пользователям;
- статус, кто обновился, а кто нет.

Результат:

- администратор видит фактическое состояние rollout по всей команде.

## Ограничения

- Все пользователи должны использовать одинаковый локальный путь папки фирмы.
- Обновление не должно выполняться поверх работающей Tekla-сессии.
- Путь `XS_FIRM` должен быть приведен к единому стандарту на всех рабочих местах.
- На первом этапе стоит поддерживать одну общую корпоративную версию стандарта для всех.

## Что не требуется учитывать

В данной схеме принимается как правило:

- пользователи не вносят локальные изменения в папку фирмы;
- конфликты пользовательских правок не рассматриваются;
- логика merge пользовательских изменений не нужна.

Это позволяет реализовать более простой и надежный controlled-update сценарий.

## Итоговый функционал коннектора

После реализации `Structura Connector` будет уметь:

- поддерживать локальную корпоративную папку фирмы Tekla;
- использовать встроенный git без ручной установки пользователем;
- проверять актуальность стандарта;
- обновлять только измененные файлы;
- автоматически применять обновление, если Tekla не запущена;
- корректно откладывать обновление, если Tekla открыта;
- уведомлять пользователя через трей;
- вести локальный журнал обновлений;
- создавать базу для дальнейшего администрирования rollout по всей команде.
