# Доступ к общей SMB-папке из любой сети через встроенный VPN (AmneziaWG)

> Технический план. **Статус: черновик до утверждения. Кодовых изменений НЕ вносить, пока план не принят.**
> Дата: 2026-06-02. Автор: подготовлено для проверки перед реализацией.

## 1. Проблема (подтверждена фактами)

Общая папка `\\62.113.36.107\BIM_Models` (SMB) работает только у пользователей из сетей, где открыт **TCP-порт 445**. У удалённых пользователей (дом, мобильный интернет) провайдер **точечно режет исходящий 445** — это массовая практика ISP (защита от SMB-червей: WannaCry/NotPetya/Conficker).

Доказано на проде:
- `Test-NetConnection 62.113.36.107 -Port 445` с домашней сети → `TcpTestSucceeded: False`, при этом `PingSucceeded: True` (сервер виден, режется именно 445).
- За 6 часов от IP удалённого пользователя (`91.105.139.7`, Дмитрий Васильев) — **ноль** попыток входа в журнале безопасности сервера (4624/4625) → его пакеты на 445 **не доходят** до сервера.
- Серверная сторона исправна: аккаунт `bim_gpsm_dv` включён, пароль валиден (проверено ручным монтированием на самом сервере), IP в firewall-allowlist на 445. Виноват **только сетевой блок 445** у провайдера.
- Из офисных сетей (`193.27.41.84`, `5.228.116.109`, `185.13.179.193`) SMB работает у 7 пользователей одновременно → сервер в порядке.

**Классический SMB нельзя перенести на другой порт** — Windows-клиент всегда идёт на 445, синтаксиса `\\server:port\share` не существует. Это подтверждает то, что говорили ранее. Обойти блок можно только туннелированием (VPN) или сменой транспорта (SMB over QUIC).

**SMB over QUIC исключён** для нашего сервера: возможность задать произвольный порт QUIC появилась только в Windows Server **2025**, а на VPS — Server **2022** (build 20348.4773), где SMB-over-QUIC жёстко на UDP **443**, а 443 уже занят Caddy (HTTP/3). Плюс QUIC требует Windows 11 у всех клиентов.

**Вывод:** единственный универсальный путь — **VPN со split-tunnel**. Внутри VPN SMB работает на родном 445, а сам туннель идёт по незаблокированному UDP-порту. Это не «SMB на другом порту», а изоляция SMB в виртуальной сети.

## 2. Цель и жёсткие ограничения

**Цель:** удалённый пользователь обновляет коннектор → VPN поднимается автоматически → `\\62.113.36.107\BIM_Models` открывается из любой сети. Без ручной установки доп. ПО, без действий пользователя.

**Ограничения (NON-NEGOTIABLE):**
1. **Не сломать работающее.** Текущий SMB на 445 для офисных, Caddy (80/443/8080), Model Sharing (9990/9991), Connector API (8080), Platform, Revit, PostgreSQL, RRAS — **не трогать**. VPN добавляется рядом, на своём порту.
2. **Не ломать другие VPN пользователя.** У людей часто активны зарубежные «обходные» VPN и/или рабочий OpenVPN. Наш VPN обязан быть **split-tunnel** (заворачивает только трафик к серверу), с редкой внутренней подсетью.
3. **Поэтапно, с проверкой и откатом.** Релиз для всех — только после подтверждения на тестовых ПК.

## 3. Почему AmneziaWG, а не WireGuard

**Стандартный WireGuard в РФ с середины 2025 не работает** через ISP с активным ТСПУ: DPI научился опознавать сигнатуру WireGuard и глушить её **независимо от порта** (смена порта не спасает). Источник: см. §9.

**AmneziaWG** — форк WireGuard с **обфускацией**: рандомизирует заголовки и размеры пакетов (параметры Jc/Jmin/Jmax/H1–H4 в 1.5; полная мимикрия под QUIC/DNS в 2.0). DPI видит «случайный UDP», блок обходится. По отзывам — держится в РФ через ТСПУ там, где обычный WG мёртв.

Критично: у AmneziaWG есть **официальная Windows-реализация** (`amnezia-vpn/amneziawg-windows`, `amneziawg-windows-client`) на базе того же кода, что WireGuard for Windows (Wintun + embeddable `tunnel.dll`). Значит:
- **сервер** можно поднять на том же Windows VPS (туннель-сервис, слушает UDP-порт, обфускация + peers);
- **клиент** встраивается в коннектор так же, как мы уже бандлим git.

Совместимость: AmneziaWG-клиент работает **на любой Windows (10/11)** — в отличие от SMB-over-QUIC, которому нужна Win11.

## 4. Архитектура решения

```
[ПК пользователя]                                  [VPS 62.113.36.107, Windows Server 2022]
 ┌─────────────────────────┐                        ┌──────────────────────────────────────┐
 │ Connector.Desktop        │                        │ AmneziaWG tunnel-service (NEW)         │
 │  ├─ bundled AmneziaWG     │  обфусц. UDP :<порт>   │  слушает UDP :<высокий порт>           │
 │  │   (tunnel.dll/клиент)  │ ─────────────────────► │  адрес внутри VPN 10.77.123.1/24       │
 │  └─ конфиг из bootstrap   │   (split-tunnel)       │  peers = устройства (из БД коннектора) │
 │                          │                        │                                        │
 │ wg-адаптер 10.77.123.X   │                        │ SMB (LanmanServer) слушает :: (445)    │
 │  AllowedIPs=10.77.123.0/24│                       │  → доступен через wg-интерфейс         │
 │  (+ 62.113.36.107/32)     │                        │  БЕЗ IP-forwarding (тот же хост)       │
 └─────────────────────────┘                        └──────────────────────────────────────┘
        │ \\10.77.123.1\BIM_Models  (или \\62.113.36.107\BIM_Models — см. §10 PoC)
        ▼
   Открывается как обычная сетевая папка; SMB на родном 445 внутри туннеля
```

**Поток данных:**
1. Коннектор по токену делает `/connect/bootstrap` → сервер возвращает (помимо SMB-кред) **персональный VPN-конфиг** (новое поле `vpn`).
2. Коннектор кладёт конфиг и поднимает туннель-сервис AmneziaWG (от админ-контекста — см. §6).
3. Клиент получает VPN-IP `10.77.123.X`. Трафик к серверу идёт по обфусцированному UDP на высокий порт.
4. SMB-папка маппится через VPN. Остальной трафик (интернет, другие VPN) — мимо нашего туннеля (split-tunnel).

**Split-tunnel (защита от конфликтов с другими VPN):**
- Клиентский `AllowedIPs` = `10.77.123.0/24` (внутренняя VPN-сеть) и, при необходимости сохранить старый путь, `62.113.36.107/32`. **НЕ** `0.0.0.0/0`.
- Внутренняя подсеть `10.77.123.0/24` выбрана редкой, чтобы не пересечься с рабочими OpenVPN (типичные `10.8.0.0/24`, `10.0.0.0/24`).
- Маршрут `/32` к серверу более специфичен, чем чужой `0.0.0.0/0`, поэтому корректно выигрывает даже при активном чужом full-tunnel VPN.

## 5. Серверная часть (разворачиваю я на VPS, изолированно)

1. **Установить AmneziaWG** на VPS (MSI/бинари `amneziawg-windows`). Отдельный продукт, не трогает существующее.
2. **Туннель-сервис сервера:** `[Interface]` с приватным ключом сервера, `Address = 10.77.123.1/24`, `ListenPort = <высокий нестандартный UDP>` (свободные подтверждены; **избегаем** 51820 как известного WG-порта, 443/UDP занят Caddy). Обфускация (Jc/Jmin/Jmax/H1–H4) — едина для сервера и клиентов.
3. **Firewall:** открыть только выбранный UDP-порт VPN (входящий). Доступ к 445 для VPN-подсети `10.77.123.0/24` — разрешить (или убедиться, что текущее правило не мешает; 445 наружу при этом НЕ открываем шире, чем сейчас).
4. **IP-forwarding НЕ требуется** — SMB на том же хосте слушает на всех интерфейсах (`::`), пакет приходит на сам сервер, маршрутизация между интерфейсами не нужна. (Проверка hairpin к публичному IP — §10.)
5. **Peers** добавляются динамически по мере подключения устройств (из БД коннектора). Механизм — см. §7.
6. **Автозапуск** туннеля как Windows-сервиса (переживает перезагрузку), как и наш Model Sharing.

**Что НЕ делаем на сервере:** не трогаем Caddy, RRAS, текущие правила 445, порты 80/443/8080/9990/9991, PostgreSQL.

## 6. Клиентская часть (bundling в коннектор — Путь A)

Переиспользуем уже существующий в проекте паттерн bundled git (`scripts/ensure_bundled_git.ps1` → `tools/` → zip в MSI → распаковка в рантайме).

1. **`scripts/ensure_bundled_amneziawg.ps1`** — скачивает официальный AmneziaWG для Windows (клиент или `tunnel.dll`+`wireguard.dll`/`wintun.dll`), кладёт в `Connector.Desktop/tools/awg/`. Вызывается из `build_msi.ps1` перед publish (как git).
2. **Подъём туннеля.** Два варианта (выбор — на PoC, §10):
   - **A1 (проще): bundled клиент + сервис.** Бандлим бинарь AmneziaWG; коннектор регистрирует туннель Windows-сервисом (аналог `wireguard /installtunnelservice <conf>`). Управление вкл/выкл из коннектора.
   - **A2 (чище для интеграции): embeddable `tunnel.dll`.** Коннектор сам регистрирует Windows-сервис с `SERVICE_SID_TYPE_UNRESTRICTED`, исполняющий туннель из `tunnel.dll` (官 способ из wireguard-windows/amneziawg-windows).
3. **Права.** Создание сетевого адаптера и Windows-сервиса требует **админ-контекста**. Установка MSI коннектора и так идёт от админа — драйвер Wintun ставится на этом этапе. Подъём/управление туннелем выполняет компонент с нужными правами (Scheduled Task под SYSTEM, как уже сделано для коннектора, либо elevated-helper). Точную модель прав фиксируем на PoC.
4. **UI:** статус VPN в коннекторе (вкл/подключено/ошибка) + лог. Кнопки не обязательны — поднимается автоматически после bootstrap.
5. **Драйвер Wintun подписан вендором** → антивирусы не блокируют (в отличие от самосборных драйверов).

## 7. Доставка VPN-конфига (через bootstrap, как SMB-доступ)

По аналогии с `get_or_create_device_access` в `connector/server/app.py`:

1. **`get_or_create_device_vpn(device_id)`**: при первом запросе генерит keypair (или принимает публичный ключ от клиента — безопаснее, приватный не покидает ПК), выделяет VPN-IP `10.77.123.X`, сохраняет в БД (новая таблица/поля), регистрирует peer на сервере (`amneziawg` set peer без перезапуска туннеля, либо перегенерация конфига сервера).
2. **`/connect/bootstrap`** возвращает новое поле `vpn`: `{ enabled, config }` (или компоненты: server pubkey, endpoint host:port, обфускация-параметры, выданный адрес, AllowedIPs). Существующие поля ответа не меняются — обратная совместимость со старыми коннекторами (они новое поле проигнорируют).
3. **Откат прост:** `vpn.enabled=false` в конфиге сервера → коннекторы VPN не поднимают, всё работает как сейчас.

DB-миграция — в обе директории (`migrations/postgres` + `migrations/sqlite`) по правилам репозитория.

## 9. Сверка с документацией (ключевые факты + источники)

- **WireGuard for Windows как сервис / embeddable:** `wireguard /installtunnelservice <conf>` и `/uninstalltunnelservice`; есть `embeddable-dll-service` (`tunnel.dll` + `wireguard.dll`), требует `SERVICE_SID_TYPE_UNRESTRICTED`. — git.zx2c4.com/wireguard-windows, github.com/WireGuard/wireguard-windows.
- **AmneziaWG обфускация и статус в РФ:** форк WireGuard-Go, рандомизация заголовков/размеров, мимикрия под QUIC/DNS; стандартный WireGuard через ТСПУ в РФ «мёртв с середины 2025», AmneziaWG держится. — docs.amnezia.org, dev.to (AmneziaWG 2.0), hub.xeovo.com.
- **AmneziaWG для Windows:** официальные `amnezia-vpn/amneziawg-windows` и `amneziawg-windows-client`, Wintun, embeddable `tunnel.dll` (build.cmd → x64\tunnel.dll), импорт `.conf`. — github.com/amnezia-vpn/amneziawg-windows(-client).
- **SMB прибит к 445, порт не меняется** (классический SMB-over-TCP). SMB-over-QUIC — отдельный транспорт, на Server 2022 только UDP 443, смена порта — c Server 2025. — learn.microsoft.com (SMB over QUIC).
- **Проверено на нашем сервере:** Server 2022 build 20348.4773; `EnableSMBQUIC` есть, но нет `*SmbServerAlternativePort*`; UDP 443 занят Caddy; SMB слушает `::`; RRAS Stopped/Disabled; высокие UDP свободны; IPEnableRouter=0.

## 10. Открытые вопросы — проверяются на PoC (Этап 1), НЕ блокируют план

1. **Путь к шаре через VPN.** Идеально сохранить `\\62.113.36.107\BIM_Models` (как в Tekla/коннекторе) — требует, чтобы доступ к публичному IP сервера через туннель работал (hairpin) + `AllowedIPs` включал `62.113.36.107/32`. Если hairpin капризничает — fallback на `\\10.77.123.1\BIM_Models` (внутренний VPN-IP) и подмену пути в коннекторе при активном VPN.
2. **AmneziaWG-сервер на Windows как multi-peer listener** — подтвердить стабильность (это форк wireguard-windows, ожидаемо умеет; проверить на PoC). Fallback при проблемах: AWG-сервер в Linux-контейнере (WSL2/Docker) на том же VPS или на отдельном Linux-VPS с маршрутом к 445.
3. **Модель прав клиента** (A1 bundled-сервис vs A2 embeddable tunnel.dll) — выбрать по результатам PoC; обе требуют админ для адаптера/сервиса.
4. **Динамическое добавление peer** (`awg set` без рестарта vs перегенерация конфига сервера) — выбрать наименее disruptive.
5. **Замер скорости** в туннеле на реальном канале.

## 12. Резюме

Единственный рабочий путь дать SMB-папку удалённым (с закрытым 445) — **VPN со split-tunnel**, и в реалиях РФ это **AmneziaWG** (обфускация против ТСПУ), у которого есть Windows-сервер и встраиваемый клиент. Бандлим клиент в коннектор (паттерн уже есть — git), конфиг отдаём через bootstrap, поднимаем автоматически. Внедряем поэтапно с проверкой на твоих ПК и откатом на каждом шаге; релиз для всех — только после подтверждения. Текущая инфраструктура и другие VPN пользователей не затрагиваются.

Файлы:
- Сервер: `connector/server/vpn.py` (provision/deprovision/peer-mgmt, под `threading.Lock`, валидация конфига), `app.py` (bootstrap-хук в try/except + deprovision при revoke/delete токена), миграции `0002_device_vpn.sql` (postgres+sqlite, `vpn_address UNIQUE`), `config.example.json` (vpn-блок).

Исправлено по ревью (ключевое): IP-allocation race (lock + UNIQUE); revoke/delete токена снимает VPN-peer; невалидный vpn-конфиг падает на сервере, а не отдаёт «мёртвый» туннель; конфиг клиента шифруется DPAPI (Enable работает после перезапуска); orphan-tunnel остаётся управляемым из UI при выключении VPN сервером; .conf с приватным ключом удаляется после установки; bootstrap не светит raw-исключение; reinstall ловит коды/таймауты; статусы через ServiceController (не sc.exe на UI-потоке); build не падает если bundling AmneziaWG не удался.

### Известные ограничения (НЕ блокеры; требуют отдельного решения/согласования)
1. **Обновление коннектора при активном VPN-туннеле** может не перезаписать `amneziawg.exe`/`wintun.dll` (заняты SYSTEM-сервисом) → возможен FilesInUse/отложенный до перезагрузки апдейт. Варианты: останавливать туннель перед апдейтом, либо вынести bundled-бинари и .conf в `%ProgramData%` (per-machine компонент). Трогает update-flow/WiX — делать отдельно, с тестом.
2. **SMB-аккаунт не деактивируется при revoke/delete токена** (`smb_user_manager.ps1` не имеет Disable/Remove) — это СУЩЕСТВУЮЩАЯ проблема (была до VPN), VPN лишь усиливает важность. Рекомендуется добавить `-Action Disable/Remove` и вызывать из revoke/delete. Трогает боевой SMB-флоу — согласовать.
3. **Приватные ключи VPN хранятся на сервере** (device_vpn.private_key + config) в открытом виде — осознанный компромисс (доверенная фирменная среда, HTTPS+токен; задокументировано в `vpn.py`). При желании — перейти на client-generated ключи или шифровать at-rest.
